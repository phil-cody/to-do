## 4. App Startup Flow

### `src/index.js`
- Imports global styles and app modules.
- Creates `defaultProject` from a default form object.
- Pushes it into `state.projects`.
- Sets `state.selectedProjectId = state.projects[0].id`.
- Calls render and event initialization functions in this order:
  1. `renderTodayDate()`
  2. `renderSidebar()`
  3. `renderDashboard()`
  4. `handlerClick()`
  5. `handlerSubmit()`
- Imports several unused `date-fns` helpers like `parseISO`, `compareAsc`, `differenceInDays`, `addDays`, `formatDistanceToNow`, `isThisWeek`, `isSameDay`.

## 5. Application State

### `src/modules/state/projects.js`
- Central state object:
  - `projects`: array of `Project` instances
  - `selectedProjectId`: id of the active project
  - `targetProjectId`: id used for editing a specific project
  - `selectedTodoId`: id of the active todo for editing/showing
  - `currentAction`: current form action type

## 6. Data Models

### `src/modules/data/Project.js`
- Instantiates a project from form data.
- Fields:
  - `project_title`
  - `project_description`
  - `todoList`: starts with one task from `defaultFormTodo`
  - `allTasks`, `overdueTasks`, `todayTasks`, `completedTasks`: computed once at construction
  - `createdAt`
  - `id`: generated by `crypto.randomUUID()`
- Uses `filterTasks.js` helpers, so current task counts are only accurate at project creation.

### `src/modules/data/Todo.js`
- Instantiates a todo/task from form inputs.
- Fields:
  - `task_title`
  - `task_description`
  - `task_due_date`: formatted date string via `date-fns`
  - `task_status`
  - `task_priority`
  - `task_createdAt`
  - `task_id`: generated by `crypto.randomUUID()`

## 7. Services Layer

### `src/modules/services/projectService.js`
- `createProject(form)`: constructs a `Project` and pushes into state.
- `getTodoListByProjectId(projectId)`: returns `todoList` from matching project.
- `getProjectById(projectId)`: returns the project object by id.
- `getAllProjects()`: returns `state.projects`.
- `updateProject(projectId, form)`: merges form values into the project.
- `deleteProject(idToDelete)`: filters state.projects to remove matching id.

### `src/modules/services/todoService.js`
- `createTodo(projectIndex, form)`: builds a `Todo` and appends to project todoList.
- `getTodoById(projectIndex, todoId)`: returns todo by its task_id.
- `updateTodo(projectIndex, todoId, form)`: merges form values into a todo.
- `deleteTodo(projectId, idToDelete)`: removes a todo by id from the project found with `findProjectIndexByID(projectId)`.
- Note: `deleteTodo` expects `projectId` but uses `findProjectIndexByID`, so its first argument is conceptually a project id, not an index.

## 8. Event Handling

### `src/modules/handlers/click.js`
- Attaches a document-level click listener.
- Handles two interaction groups:
  1. `.project-item` clicks for selection
  2. buttons with `data-btn` actions
- Project selection branch:
  - Sets `state.selectedProjectId` from the clicked `.project-item`
  - Calls `renderSidebar()` then `toggleSelectedProject()` then `renderDashboard()`
- Button actions include:
  - add project/task dialog open
  - edit project/task dialog open
  - delete project/task
  - close dialog
  - show full task details
- Project delete logic:
  - if deleted project is active, sets `state.selectedProjectId = state.projects[0].id`
  - calls `toggleSelectedProject()` without id in that path
  - deletes the project
  - rerenders dashboard and sidebar
- This file contains commented-out old deletion logic, indicating active debugging and iteration.

### `src/modules/handlers/submit.js`
- Handles submit on `#addProjectForm` and `#addTaskForm`.
- On project form submit:
  - if `EDIT_PROJECT`, updates existing project by `state.targetProjectId`
  - if `ADD_PROJECT`, creates a new project
  - sets `state.selectedProjectId` to the newly created or last project id
  - rerenders sidebar/dashboard and clears the form
- On task form submit:
  - if `EDIT_TASK`, updates existing todo
  - if `ADD_TASK`, creates a new todo
  - rerenders sidebar/dashboard and clears the form
- Note: when editing a project, the code still sets selection to `state.projects.at(-1).id`, which changes selection to the last project even if another project was edited.

## 9. UI Rendering

### `src/modules/UI/renderSidebar.js`
- Clears the `.projects` container.
- For each `state.projects`, renders a `projectCard` and appends it.
- Adds `selected` class when `project.id === state.selectedProjectId`.
- Imports `toggleSelectedProject` but does not use it.

### `src/modules/UI/renderDashboard.js`
- Computes `currentProjectIndex` by finding project index of `selectedProjectId`.
- Clears `.dashboard` and re-creates a container with header and todo list.
- Uses `state.projects[currentProjectIndex]` to render title, description, and todo cards.
- If project is missing, renders placeholder text.
- Note: if `state.selectedProjectId` is `null` or invalid, `findProjectIndexByID` returns `-1`, and `state.projects[-1]` is `undefined`.

### `src/modules/UI/renderProjectCard.js`
- Builds a project card DOM tree with title, counts, and edit/delete buttons.
- Imports `TODO_STATUS` and `TODO_PRIORITY` but does not use them.

### `src/modules/UI/renderTodoShort.js`
- Builds a compact todo card with title, description, due date, status select, priority select, and action buttons.
- Formats the due date using `isToday`, `isTomorrow`, or `format`.
- Sets `<option selected>` for current status and priority.

### `src/modules/UI/renderTodoFull.js`
- Builds a dialog display for a single todo.
- Includes close button, details, status and priority selects, created date, and action buttons.
- Uses `format`, `isToday`, and `isTomorrow`.

### `src/modules/UI/renderTodayDate.js`
- Updates the `.date` element every second using `setInterval`.
- Formats time and date with `date-fns`.

### `src/modules/UI/clearSection.js`
- Removes all child nodes from a DOM container.

## 10. Utility Modules

### `src/modules/utils/constants.js`
- Defines constants for:
  - todo status values
  - todo priority values
  - dashboard view modes
  - button dataset actions

### `src/modules/utils/defaultTodo.js`
- Creates a default `FormData` object used by `Project` construction.
- Default task fields:
  - title: "Add first task"
  - description: "What is the first thing you need to do?"
  - due date: `2026-01-10`
  - status: `completed`
  - priority: `low`

### `src/modules/utils/closeDialogs.js`
- Closes every `<dialog>` currently in the DOM.

### `src/modules/utils/clearForm.js`
- Clears text and date inputs inside a form.
- Note: only empties fields of type `text` and `date`.

### `src/modules/utils/findIndexById.js`
- `findProjectIndexByID(currentId)`: returns index of project with matching id.
- `findTaskIndexByID(currentId)`: returns index of todo within currently selected project.
- Note: this helper assumes `state.selectedProjectId` is valid before use.

### `src/modules/utils/filterTasks.js`
- `filterOverdue(date, status)` returns true if a task is overdue and not completed.
- `filterToday(date)` returns true if a due date is today.
- `filterCompleted(status)` returns true if status equals completed.

### `src/modules/utils/toggleSelectedProject.js`
- Walks through `document.querySelector('.projects').children` and toggles the `selected` class.
- Works only on existing DOM elements.
- Because `renderSidebar()` reconstructs the list, this helper can be redundant and fragile.

### `src/modules/utils/addTaskInProject.js`
- Empty file. Present in the tree but unused by the app.

## 13. Key Behavior Patterns

### Selection flow
- `renderSidebar()` marks current project card as selected when `project.id === state.selectedProjectId`.
- `handlerClick()` also calls `toggleSelectedProject()` after rerendering sidebar, which is redundant.
- Because sidebar is reconstructed, selection should be applied during render, not afterward.

### Project deletion
- `handlerClick()` updates `selectedProjectId` if the deleted project is active.
- It then calls `deleteProject()` and rerenders dashboard/sidebar.
- Edge cases are not fully protected: deleting the only project can leave `selectedProjectId` invalid and `renderDashboard()` may see `-1` index.

### Editing flow
- Project editing uses `state.targetProjectId` separate from `selectedProjectId`.
- Task editing uses `state.selectedTodoId`.
- After editing or adding a project, selection always jumps to the last project, even on edit.

## 14. Known Issues and Code Smells

- `renderSidebar()` imports `toggleSelectedProject` but does not use it.
- `renderProjectCard.js` imports unused constants.
- `src/index.js` imports many unused `date-fns` helpers.
- `Project` summary counters are static and not recalculated after todo updates.
- `renderDashboard()` may access `state.projects[-1]` when `selectedProjectId` is invalid.
- `deleteTodo()` has a misleading parameter name (`projectId` but uses the helper to find index), increasing risk of misuse.
- `toggleSelectedProject()` depends on existing DOM structure and is fragile after rerender.
- `clearForm()` only resets text/date and leaves other input types unchanged.
- `defaultTodo` hardcodes a past due date (`2026-01-10`), which may make the default todo appear overdue depending on current date.
- `addTaskInProject.js` is an orphaned, empty module.

## 15. Knowledge Graph

### Core data path
- `src/index.js` creates initial project and selects it.
- `Project` uses `defaultTodo` and `filterTasks`.
- `Todo` formats dates during construction.
- `state.projects` holds all project objects.

### Interaction path
- User clicks a `.project-item` → `handlerClick()` updates `state.selectedProjectId` → `renderSidebar()` and `renderDashboard()` redraw UI.
- User submits a form → `handlerSubmit()` creates/updates models → rerenders sidebar/dashboard.
- User deletes a todo/project → service functions mutate `state.projects` → `handlerClick()` rerenders UI.

### UI rendering path
- `renderSidebar()` uses `renderProjectCard()`.
- `renderDashboard()` uses `renderTodoShort()`.
- `SHOW_TASK` action uses `renderTodoFull()` in a dialog.
- `renderTodayDate()` keeps the header clock live.

## 16. Summary of Changes and Observations

- The project is structured as a simple stateful SPA with manual DOM rendering.
- The main logic is split into state, services, handlers, UI, and utilities.
- Some utility and render code are duplicated or redundant (
  `toggleSelectedProject()` vs `renderSidebar()` selection handling).
- There are clear areas for state-safety hardening, especially around deletion and invalid selection.
- `project-knowledge-graph.md` now reflects the actual source files and their current implementation.
